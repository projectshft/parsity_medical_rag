Week 1: Introduction to RAG

The Problem

Large Language Models are powerful, but they have a critical limitation:

LLMs don't know your private data.
GPT-4 has never seen your company's documents, your patient records, or your codebase.

What happens when you ask about your data?

You ask LLM responds Problem
"What medications is John Smith taking?" "I don't have access to patient records..." No access to your database
"Summarize our Q3 sales report" Makes up plausible-sounding numbers Hallucination
"Find bugs in our auth system" Generic security advice Doesn't know your code

The Solution: RAG

Retrieval-Augmented Generation = Give the LLM your data at query time.

┌──────────────────┐ │ User Question │ └────────┬─────────┘ │ ▼ ┌──────────────────┐ ┌──────────────────┐ │ RETRIEVE │────▶│ Your Data │ │ Find relevant │ │ (database, │ │ information │ │ documents) │ └────────┬─────────┘ └──────────────────┘ │ ▼ ┌──────────────────┐ │ AUGMENT │ │ Add context to │ │ the prompt │ └────────┬─────────┘ │ ▼ ┌──────────────────┐ │ GENERATE │ │ LLM creates │ │ response │ └──────────────────┘

The RAG Formula

prompt = system_instructions + retrieved_context + user_question

# Example:
"""
You are a medical assistant.

PATIENT DATA:
- John Smith, 45, diabetic
- Current medications: Metformin 500mg, Lisinopril 10mg
- Recent A1C: 8.2%

USER QUESTION: What medications is John Smith taking?
"""

Why Hybrid RAG?

Medical data comes in two forms:

Structured Data Unstructured Data
  • Patient names, DOB
  • Diagnosis codes (ICD-10)
  • Lab values (A1C = 8.2)
  • Medication lists
  • Clinical notes
  • "Patient reports chest pain..."
  • "Discussed lifestyle changes..."
  • Doctor observations
Best tool: SQL Best tool: Vector Search
Hybrid RAG uses both:
  • SQL for precise queries ("patients with A1C > 9")
  • Vector search for semantic queries ("breathing problems")

What is Vector Search?

Vector search finds semantically similar content, not just keyword matches.

The Process

  1. Embed: Convert text to numbers (vectors)
  2. Store: Save vectors in a vector database
  3. Query: Convert question to vector, find nearest neighbors
# Text → Vector (embedding)
"patient has trouble breathing" → [0.12, -0.34, 0.56, ...]

# Similar meanings = similar vectors
"dyspnea"                       → [0.11, -0.33, 0.57, ...]  # Very close!
"headache"                      → [0.82, 0.15, -0.23, ...]  # Far away

Why this matters

Query Keyword Search Vector Search
"breathing problems" Only finds "breathing" Finds: dyspnea, shortness of breath, respiratory distress
"heart issues" Only finds "heart" Finds: cardiac, cardiovascular, chest pain, arrhythmia

Our Architecture

┌─────────────────────────────────────────────────────────────┐ │ User Query │ │ "Find diabetic patients with notes about foot pain" │ └─────────────────────────────────────────────────────────────┘ │ ▼ ┌─────────────────────────────────────────────────────────────┐ │ Query Analyzer (LLM) │ │ "diabetes" → structured "foot pain" → semantic │ └─────────────────────────────────────────────────────────────┘ │ │ ▼ ▼ ┌─────────────────────────┐ ┌─────────────────────────┐ │ Neon (PostgreSQL) │ │ Pinecone │ │ │ │ │ │ SELECT * FROM patients │ │ Search clinical notes │ │ JOIN conditions │ │ for "foot pain" │ │ WHERE code = 'diabetes' │ │ filtered to diabetics │ └─────────────────────────┘ └─────────────────────────┘ │ │ └────────────┬────────────┘ ▼ ┌─────────────────────────────────────────────────────────────┐ │ LLM │ │ Generates response from merged SQL + vector results │ └─────────────────────────────────────────────────────────────┘

The Data: Synthea

Synthea generates realistic synthetic patient records in FHIR format.

What's in a patient bundle?

  • Patient: Name, DOB, gender, address
  • Conditions: Diagnoses (diabetes, hypertension, etc.)
  • Observations: Lab results, vitals
  • MedicationRequest: Prescriptions
  • DocumentReference: Clinical notes (SOAP format)

SOAP Notes

S (Subjective):  What the patient reports
O (Objective):   What the doctor observes/measures
A (Assessment):  Diagnosis and clinical reasoning
P (Plan):        Treatment plan
Key insight: The clinical notes contain valuable information that doesn't fit in structured fields. That's why we need vector search.

Discussion: Metadata

When we chunk and embed clinical notes, what metadata should we store?

Think about:

  • How would you filter search results?
  • What context does the LLM need?
  • What queries might users ask?

Possible metadata:

  • patientId - Link back to structured data
  • date - When was this note written?
  • type - SOAP section? Visit type?
  • provider - Which doctor?
  • What else?
