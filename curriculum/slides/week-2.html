An agent that decides — then answers from the records

The setup: two engines, no driver

After Week 1 you have two ways to find things. The database answers exact facts — "how many patients have diabetes?" is a COUNT. The vector store answers by meaning — "notes that describe shortness of breath" finds the one that says "dyspnea."

But a user doesn't know which engine they're hitting. They just type. Something has to read the question and choose — and choosing wrong means a right-sounding, wrong answer.

Say it plainly: Week 1 built the engines. This week builds the driver. The whole session is: how does software decide which tool a plain-English question needs?

The move: let the model classify, then route in code

We don't ask the model to answer the medical question. We ask it one narrow thing first: what kind of question is this, and what is it about? Then plain code routes to the right engine and hands the model only the records that came back.

The model's job

Read the question → return a small, typed label: the intent, the conditions/names mentioned, which engines are needed.

Your code's job

Trust that label to run the database, the vector search, or both — then answer only from what came back.

Design point to land: the LLM is a router and a writer, never the source of facts. Splitting "decide" from "answer" is what makes this debuggable.

Turning the model into a typed function

Routing only works if the model's answer has a fixed shape you can branch on — not a paragraph you have to parse. So we hand it a schema and make it fill in the blanks:

const QueryAnalysis = z.object({
  intent: z.enum(['population_analytics', 'clinical_note_search',
                  'hybrid_query', 'patient_summary', /* … */]),
  requiresSQL:    z.boolean(),
  requiresVector: z.boolean(),
  semanticQuery:  z.string().nullable(),  // expanded terms for meaning search
});

The schema is the contract. The model can only answer inside these fields — so your if (requiresSQL) is always safe to write.

💻 Code walkthrough

The analyzer: analyzeQuery()

One call. A system prompt describing the two stores, the schema as the output format, and temperature: 0 so the same question routes the same way every time.

const response = await client.responses.parse({
  model: 'gpt-4o-mini',
  input: [
    { role: 'system', content: SYSTEM_PROMPT },   // "you route medical queries…"
    { role: 'user',   content: userQuery },
  ],
  temperature: 0,
  text: { format: zodTextFormat(QueryAnalysis, 'queryAnalysis') },
});

return QueryAnalysis.parse(response.output_parsed);  // typed + validated

Three paths out of the analyzer

                    analyzeQuery(question)
                            │
         ┌──────────────────┼──────────────────┐
         ▼                  ▼                  ▼
   requiresSQL only   requiresVector only   both
   ───────────────    ─────────────────    ────────────
   "how many have     "notes describing    "notes about sleep
    diabetes?"         shortness of breath"  for diabetics"
         │                  │                  │
     database            vector store       SQL then vector
     COUNT / filter      meaning search     (hybrid)

The flags requiresSQL and requiresVector pick the branch. Two booleans decide the entire route.

💬 Discussion

Which path — and what could route it wrong?

  • "How many patients have high blood pressure?"
  • "Which notes mention trouble sleeping?"
  • "What do the notes say about coping for patients with depression?"
  • "Give me a summary of Abe Frami"
  • "Which patients are on insulin?"
Facilitate: the last one is a trap — sounds like a clean count, but medication filtering isn't wired to the database yet, so the honest answer is "the router wants SQL, but there's no med filter behind it." Great lead-in to Week 4's analyzer evals. Don't sell it as a working demo.

Hybrid: facts narrow the world, meaning ranks what's left

Some questions need both. "What do the notes say about sleep for patients with depression?" The word "depression" is an exact fact — the database knows exactly who has it. "About sleep" is meaning — only the notes know that.

1.  SQL:    get the patient IDs who have the condition   → [id, id, …]
2.  Vector: search "sleep quality insomnia rest" — but ONLY
            within those patients' notes
            └─ the fact is a filter on the meaning search

Order matters: narrow first, then rank. Searching all 144k notes and hoping the depression ones float up is slower and leaks everyone else in.

Say it plainly: the SQL result isn't the answer — it's the filter the vector search runs inside. Hold this; the privacy bug on slide 14 is exactly this filter going empty.

The grounding contract

Once the records come back, the model writes the answer — under a strict rule baked into its system prompt:

  • Answer only from the retrieved records
  • If the information isn't there, say so — don't infer, don't fill the gap
  • Never invent a diagnosis, a med, or a number

A confidently wrong medical assistant is worse than none. "I don't have that in the records" is a correct answer.

Design point to land: retrieval is what earns the model the right to speak. No record → no claim. This is the line between a demo and something you'd let near a chart.

The tell: what happens when the answer isn't there

Ask the agent something plausible that the data simply doesn't contain — a detail no note or record holds. Two possible behaviors:

✅ Grounded

"I don't see that in the records." It retrieved nothing relevant and refused to guess.

❌ Confabulated

A smooth, specific, invented answer. Reads great. Is fiction. This is the failure that gets someone hurt.

You can't tell which you built by reading the happy path. You find out by feeding it bait.

🛠 Mini-challenge

Run it — then try to make it lie

  • Run one question per path: a count (diabetes), a note search (shortness of breath), a hybrid (sleep + depression). Read the routing JSON each time.
  • Bait it: ask something the records can't hold. Did it refuse — or did it write fiction?
  • Break the filter: send a hybrid query whose condition matches no one. Watch whose notes come back.
The point isn't that it works — it's building the reflex to attack your own system. A grounded refusal and an empty-filter leak both hide until you go looking. We measure this properly in Week 4.

When the filter narrows to nobody

Back to that hybrid line. If the condition matches zero patients, patientIds is an empty list — and patientIds?.length ? … : undefined quietly passes undefined, which means no filter:

patientIds = []            // nobody matched the condition
patientIds?.length         // 0 → falsy
  ? patientIds
  : undefined              // → search ALL 144k notes, every patient

"Zero patients" should mean zero notes. Instead it means everyone's notes. The empty filter doesn't fail loudly — it silently widens to the whole corpus. That's a privacy leak, not a ranking bug.

Say it plainly: an empty filter that means "match everything" is one of the most common real RAG security bugs. The fix is to treat empty-narrowing as return-nothing, not fall-through. We fix it live in the break-it bank.
