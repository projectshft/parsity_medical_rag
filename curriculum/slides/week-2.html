The setup: two engines, no driver

After Week 1 you have two ways to find things. The database answers exact facts — "how many patients have diabetes?" is a COUNT. The vector store answers by meaning — "notes that describe shortness of breath" finds the one that says "dyspnea."

But a user doesn't know which engine they're hitting. They just type. Something has to read the question and choose — and choosing wrong means a right-sounding, wrong answer.

Say it plainly: Week 1 built the engines. This week builds the driver. The whole session is: how does software decide which tool a plain-English question needs?

The move: hand the model the schema, run what it writes

The naive way is to hand-code a query builder for every shape of question — one for counts, one for filters, one for "the youngest." You write those forever. Instead, we describe the database schema to the model and let it write one read-only SELECT. Plain code runs it and answers only from the rows.

The model's job

Read the question + the schema → write one SELECT. Superlatives, combined filters, "average age" — it writes the WHERE/ORDER BY itself.

Your code's job

Check it's a read-only SELECT, run it, answer only from the rows — never the model's own memory.

Design point to land: this is "text-to-SQL." It kills the query-builder treadmill — but it moves the hard part to two places: keeping it safe, and grounding it in the real vocabulary. Those are the next two slides.

Guardrail #1 — a typed function, and only a SELECT

Same structured-output trick as yesterday — but now the payload is the query itself. We hand the model a schema and it fills in the blanks:

const SqlQuery = z.object({
  sql:         z.string(),   // one read-only SELECT — no INSERT/UPDATE/DELETE
  explanation: z.string(),   // one line: what this returns
});

An LLM writing SQL is an injection surface. Two layers stop it: a validator that accepts only a single SELECT (assertReadOnly), and — in production — a database role that is physically read-only, so a bad query can't write even if the validator is fooled.

Belt and suspenders: the validator makes the common case cheap; the read-only role makes the worst case impossible.

💻 Code walkthrough

The SQL agent: textToSqlQuery() — and guardrail #2

The system prompt is the schema (tables + columns) plus a sample of real values. The model returns { sql }; we validate it, then run it read-only.

const { sql } = await client.responses.parse({ /* SqlQuery, temp 0 */ });
assertReadOnly(sql);                       // SELECT-only, no ';'
const rows = await prisma.$queryRawUnsafe(sql);   // read-only role

Guardrail #2 — grounding. The schema tells the model a display column exists, not what's in it. Ask for "smokers" and it writes ILIKE '%smoker%' → zero rows, because the value is literally "Smokes tobacco daily". You fix it by feeding the model the real vocabulary.

lib/text-to-sql.ts. The schema is structure; grounding is meaning — text-to-SQL needs both.

Facilitate: run it live (next section). Read the SQL it wrote aloud — that's the agent thinking on paper. Then run the "heart attack" question and watch it return 0 rows for the wrong reason.

The router + two agents + an aggregator

The SQL agent is one of two specialists. A tiny router (analyzeQuery) reads the question and sets two booleans — which agents to run — and they run in parallel:

              analyzeQuery(question)   ← the router
                        │
        ┌───────────────┴───────────────┐
        ▼ requiresSQL          requiresVector ▼
   SQL AGENT (text-to-SQL)      VECTOR AGENT (searchClinicalNotes)
        └───────────────┬───────────────┘
                        ▼
                 AGGREGATOR (an LLM merges both, streams the answer)

The router is dumb on purpose — two booleans, no query-building. All the judgment lives in the SQL agent's schema + the aggregator's grounding rule.

💬 Discussion

Which engine — and what could the SQL agent get wrong?

  • "How many patients have high blood pressure?"
  • "Which notes mention trouble sleeping?"
  • "What do the notes say about coping for patients with depression?"
  • "Who is the youngest patient with diabetes?"
  • "Any patients who had a heart attack?"
Facilitate: the last one is the trap — the SQL agent writes a clean ILIKE '%heart attack%' and returns 0 rows, but the records DO have them, stored as "Myocardial Infarction." A confident, wrong "none." That's the semantic-grounding gap — name it, don't fix it live; it's the lead-in to Week 4's evals. ("Youngest with diabetes" works — a superlative the old query builders never gave for free.)

Hybrid: facts narrow the world, meaning ranks what's left

Some questions need both. "What do the notes say about sleep for patients with depression?" The word "depression" is an exact fact — the database knows exactly who has it. "About sleep" is meaning — only the notes know that.

1.  SQL:    get the patient IDs who have the condition   → [id, id, …]
2.  Vector: search "sleep quality insomnia rest" — but ONLY
            within those patients' notes
            └─ the fact is a filter on the meaning search

Order matters: narrow first, then rank. Searching all 144k notes and hoping the depression ones float up is slower and leaks everyone else in.

Say it plainly: the SQL result isn't the answer — it's the filter the vector search runs inside. Hold this; the privacy bug on slide 14 is exactly this filter going empty.
💻 Code walkthrough

Hybrid, built by hand

Narrow first, then rank. The SQL agent returns the matching patients' ids; the vector search runs inside that set:

// 1. exact — ask the SQL agent for the ids
const { rows } = await textToSqlQuery(`the patient ids of ${condition}`);
const patientIds = rows.map(r => String(r.id));

if (patientIds.length === 0) return [];   // ← watch this guard

// 2. fuzzy — rank their notes, filtered to those ids
const notes = await searchClinicalNotes(semanticQuery, { topK: 10, patientIds });

The patientIds filter is a real feature of searchClinicalNotes. That empty-list guard is doing something load-bearing — we'll break it on purpose later.

The grounding contract

Once the records come back, the model writes the answer — under a strict rule baked into its system prompt:

  • Answer only from the retrieved records
  • If the information isn't there, say so — don't infer, don't fill the gap
  • Never invent a diagnosis, a med, or a number

A confidently wrong medical assistant is worse than none. "I don't have that in the records" is a correct answer.

Design point to land: retrieval is what earns the model the right to speak. No record → no claim. This is the line between a demo and something you'd let near a chart.

The tell: what happens when the answer isn't there

Ask the agent something plausible that the data simply doesn't contain — a detail no note or record holds. Two possible behaviors:

✅ Grounded

"I don't see that in the records." It retrieved nothing relevant and refused to guess.

❌ Confabulated

A smooth, specific, invented answer. Reads great. Is fiction. This is the failure that gets someone hurt.

You can't tell which you built by reading the happy path. You find out by feeding it bait.

🛠 Mini-challenge

Run it — then try to make it lie

  • Run one question per path: a count (diabetes), a note search (shortness of breath), a hybrid (sleep + depression). Read the routing JSON each time.
  • Bait it: ask something the records can't hold. Did it refuse — or did it write fiction?
  • Break the filter: send a hybrid query whose condition matches no one. Watch whose notes come back.
The point isn't that it works — it's building the reflex to attack your own system. A grounded refusal and an empty-filter leak both hide until you go looking. We measure this properly in Week 4.

When the filter narrows to nobody

Back to that guard. If the condition matches zero patients, patientIds is an empty list — and searchClinicalNotes treats "empty" as "no filter given":

const patientIds = [];     // nobody matched the condition

// inside searchClinicalNotes:
if (patientIds && patientIds.length > 0) { /* filter to them */ }
// else → NO filter → search every patient's notes

"Zero patients" should mean zero notes. Instead it means everyone's notes. Drop the empty-list guard and the search silently widens to the whole corpus — a privacy leak, not a ranking bug.

Say it plainly: an empty filter that means "match everything" is one of the most common real RAG security bugs. The fix is the guard on the last slide: empty-narrowing returns nothing, never falls through. We break it and fix it live in the break-it bank.
