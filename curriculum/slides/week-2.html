The question every corpus forces on you

Meaning-based search matches a user's question against stored pieces of text, and hands the best pieces to the LLM. So someone has to answer, for every corpus:

What is one piece?

Too big: a 40-page document "mentions" diabetes, sleep, surgery, and billing — a mediocre match for all of them. Too small: "And he said unto them" matches confidently and tells you nothing. Said what? Who? To whom?

Say it plainly: chunking is just splitting documents into retrieval-sized pieces. The whole week is "how big, and where do you cut?"

The move: decide from the data, not from habit

Most tutorials make chunking "step 2 of every pipeline." That's the trap. Whether you chunk at all is a property of your corpus — and you find it by measuring.

Our clinical notes

~144,000 notes · ~450 chars each · one note = one topic, one date. Already a piece. Chunking it would only break it.

The King James Bible

1 document · 4.3 million chars. Unusable as one piece. Must be chunked.

Design point to land: we learn chunking on the Bible because our notes don't need it. The contrast is the lesson — "no metric, no decision," applied to architecture.
💬 Discussion

Chunk it or not? Decide from the number

  • Clinical notes — avg 450 chars, max ~3,000
  • Support-ticket transcripts — avg 80,000 chars
  • Tweets — avg 120 chars
  • Notes again — but one in a thousand is a 40,000-char discharge summary
Facilitate: push them off "the average." The last one is the trap — an average can hide a monster. The rule isn't "is the average small," it's "is every document an acceptable piece."

The two ways a piece is wrong

TOO BIG                          TOO SMALL
─────────────────────            ─────────────────────
one chunk covers                 "And he said unto them"
many topics                      → complete sentence,
→ matches everything             → zero standalone meaning
  a little, nothing well         → confident match, useless

       the sweet spot: one self-contained thought,
       big enough to mean something on its own

There is no universal right answer. "Just use 512 with 50 overlap" is reciting a default, not making a decision. The week is about how to make the decision.

💻 Code walkthrough

Get the corpus, read its shape

mkdir -p data/bible
curl -o data/bible/kjv.txt \
  https://www.gutenberg.org/cache/epub/10/pg10.txt
head -120 data/bible/kjv.txt

Three things to point at live: a Gutenberg license header that is NOT scripture (must be stripped), book titles as plain lines, and every verse prefixed chapter:verse like 1:1 In the beginning… — built-in structure that pays off Wednesday.

💻 Code walkthrough

The simplest chunker: slice every 500 chars

npm run bible:fixed

Corpus: 4,307,701 characters
Chunks: 8,616 (size=500, overlap=0)

Start of chunk 1000: uses of the cities of the Levites…

A fixed window doesn't know what a word is. Chunk 999 ended on ho; chunk 1000 starts on uses — the word "houses" torn in half. Neither side owns it.

Don't squint — measure

npm run bible:audit -- data/bible/chunks-fixed.jsonl

chunks:               8,616
size min/median/max:  201 / 500 / 500 chars
starts mid-word:      7,634 (88.6%)
ends mid-sentence:    8,339 (96.8%)
has metadata:         0 (0.0%)

"The chunks feel bad" starts arguments. "88.6% start mid-word" ends them. And metadata: 0% — a bill that comes due Thursday.

Say it plainly: a chunk is what gets matched AND what gets read. A broken edge doesn't just rank worse — the model fills in the torn-off half and misleads.
💬 Discussion

"I'll just pick a better size." Will that fix it?

We can re-run at --size 200 or --size 2000. Predict before we run: what happens to the 88.6% mid-word rate?

Facilitate: let them bet, then run it. The percentages barely move — 85–90% at every size. The flaw isn't the number; it's that character positions don't align with meaning, at any size. The knob you were given doesn't control the thing that's broken.

The fix: let the document tell you where it bends

The Bible arrives pre-jointed — verses are marked, books are titled. So instead of cutting every 500 chars, pack whole verses:

  • add verses until the chunk reaches a target (~500 chars)
  • never split a verse — overshooting the target is fine
  • never cross a book boundary — Malachi's end and Matthew's start aren't one piece
Design point to land: use the strongest structure the corpus actually has. Generic separator-splitting is for corpora with no real joints — this one declares its own.

Overlap: insurance for the seams

chunk 0:  verses 1:1 … 1:6
                       └── 1:6 repeated ──┐
chunk 1:                  verses 1:6 … 1:11

A question about "the firmament dividing the waters"
might need 1:6 AND 1:7 together. Repeat the seam verse
so whichever chunk is retrieved, the thought is whole.

Cost is real: overlap duplicates text → more storage, more near-duplicates crowding results. It's a dial, not a virtue. We set it to 1 verse and let the audit say what we bought.

💻 Code walkthrough

Chunk on structure, then compare the audits

npm run bible:smart        # Verses: 31,081 · Chunks: 9,737

fixed (Tue)

mid-word starts 88.6%
ends mid-sentence 96.8%
has metadata 0%

structure-aware (Wed)

mid-word starts 0.0%
ends mid-sentence 3.5%
has metadata 100%

We paid for it: max chunk is now 1,675 chars (the longest verse, Esther 8:9, ships whole). We bought integrity and citability with size variance. Perfection isn't on the menu — measured improvement is.

Metadata: the part everyone skips

A chunk without metadata is a quote without a source. The text gets matched; the metadata does the three jobs text alone can't:

Cite    "Where did this come from?"   → reference: "Psalms 48:10"
Filter  "Search only in Psalms"       → book == "..." (a field)
Debug   "Why did THIS chunk match?"   → find it in the source
Design point to land: in our system book is patientId. Getting the filter wrong isn't a bad result — it's showing one patient's chart inside another patient's answer. Decide metadata at chunking time; afterward, provenance is gone.

The five chunking failure modes — each with a number

  • 1. Fragmentation — cuts mid-word/sentence → % starting mid-word
  • 2. Context orphaning — whole sentence, no meaning alone → min size, sampling
  • 3. Boundary straddling — answer spans a seam → overlap setting
  • 4. Anonymity — can't cite / filter / trace → % with metadata
  • 5. Wrong granularity — too big or too small → size vs the corpus's unit

They trade against each other: never-split-a-verse killed fragmentation but created size variance. You don't solve chunking — you balance dials, for this corpus, with measurements.

🛠 Mini-challenge

Make a failure mode appear on demand

  • Rebuild the smart chunks with zero overlap: npm run bible:smart -- --overlap-verses 0
  • Look at two adjacent chunks at a seam — they now share no verse
  • Write a question whose answer needs the last verse of one AND the first verse of the next. That question is now unanswerable from any single chunk.
  • Restore overlap to 1 afterward — and audit to prove the seam coverage came back
The point isn't the Bible — it's holding boundary-straddling in your hands so you recognize it in any corpus.

Go find out

  • The most common general-purpose chunker splits on a cascade of separators (paragraphs → sentences → words). When is that the right choice — and why did we reject it for the Bible?
  • Some teams prepend generated context to each chunk before storing it. What problem is that solving that metadata fields alone can't?

Bring one finding back. No single right answer — defend your reasoning with the five-mode vocabulary.
