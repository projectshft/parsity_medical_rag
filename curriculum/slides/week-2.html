Metadata: the jobs the text can't do

The note's text is what gets matched by meaning. The metadata riding on each vector is everything else a real system must do with a match:

Filter

"Search only this patient's notes." One patient averages ~105 notes (the heaviest chart has 1,632) — without the filter you always search all 21,090.

Cite & debug

"Whose note is this, and why did it match?" Without metadata a hit is an anonymous fragment you can't trace back.

The filter runs inside index.query — the index finds the top-K among only the matching vectors. Filtering afterward gives you "the global top 10 minus strangers" — often zero of the right patient's notes.

Design point to land: in-query vs post-filter is both a quality bug and a safety bug. Quality: post-filtering can return nothing relevant. Safety: post-filtering means strangers' note content leaves the database on every query. Foreshadow slide 7 — we'll make the leak visible. Filters are exact-match ($in compares whole strings); meaning lives in the text, facts in the metadata.
💻 Code together

Implement searchClinicalNotes

You know every ingredient already. The recipe:

1. Embed the query            — createEmbedding(query)  (already imported)
2. Build the filter, only if patientIds was given:
     one id   → { patientId: id }
     several  → { patientId: { $in: ids } }
     none     → no filter at all  (not an empty object!)
3. Query the index            — index.query({ vector, topK,
                                  includeMetadata: true, filter })
4. Map each match             → { id, score, patientId,
                                  patientName (firstName + lastName metadata),
                                  contentPreview (truncated content) }

Write it before peeking at any solution. filter: {} and filter: undefined are different things; forget includeMetadata: true and every field comes back undefined.

Pace: ~15 min heads-down, you live-code it last on screen. The two planted potholes are on the slide (empty filter object, missing includeMetadata) — let someone hit them before naming them. Project visuals/vector-search.html while people finish.

When cosine lies

query:  "the patient reports shortness of breath"

0.61   "the patient reports knee pain"      ← template match wins
0.58   "dyspnea on exertion"               ← the actual meaning, buried

An embedding summarizes each text separately, before any query exists. At query time all you can do is compare two pre-made summaries — anything lost in compression is lost forever. Sentences that share a template can out-score sentences that share a meaning.

A reranker has the one luxury embeddings can't: it reads the query and the document together, at query time, and scores how well this document answers this query.

Ask first: "has anyone already seen a wrong note ranked #1?" Someone has — use their example. Then the precise version: pre-compression vs reading-the-pair. Project visuals/reranking.html here or on the next slide.

The funnel: cheap-and-broad, then expensive-and-careful

query ──▶  vector search   ── top 25 ──▶   reranker    ── top 5 ──▶  results
           fast, broad                     slow, careful
           scans ~21k vectors              full pass per query+doc pair
  • Why not rerank all 21,090? A reranker reads every candidate against the query — you can't afford that corpus-wide, per query.
  • The over-fetch is the point: a great note ranked #19 by cosine is invisible unless the funnel mouth is wide enough to include it.
  • No new vendor: Pinecone hosts the reranker (bge-reranker-v2-m3) on the same PINECONE_API_KEY you already have.

Same logic as a recruiter skimming 200 résumés and carefully interviewing 6. You'll meet this funnel shape again and again.

One more warning to plant now: rerank scores are normalized 0–1 but they are not cosines — comparing a 0.91 rerank score to a 0.58 cosine is meaningless. Order within one list is all that transfers.

Code can't branch on prose

Everything so far ends in text for a human. The next component's output is consumed by code — and if the model replies "this looks like a question about a specific patient…", what does your if statement test? You need:

{ "requiresSQL": true, "requiresVector": false, "semanticQuery": null }

1. Define a Zod schema        — .describe() on every field (descriptions ARE prompting)
2. Infer the type             — z.infer<typeof Schema>
3. openai.responses.parse()   — with zodTextFormat(Schema, 'name'), temperature 0
4. Schema.parse(output_parsed) — trust, but parse

"Please respond in JSON" is a request — fences, preambles, and invented enum values all show up eventually. A schema is a constraint: the model can't produce anything else.

Keep this brisk (~7 min) — it's a pattern, not a project, and it powers everything from here. Two beats to land: a required field on absent information doesn't error, it fabricates (make schemas match reality with nullable()); and step 4 isn't redundant — it turns SDK drift/refusals/truncation into one loud error at the call site.
💻 Code together

The selector — your first agent

It answers exactly one question: which store(s) does this question need? The whole schema:

const PlanSchema = z.object({
  requiresSQL:    z.boolean().describe('Structured data — counts, filters, a specific patient.'),
  requiresVector: z.boolean().describe('The notes — what they say / describe / mention.'),
  semanticQuery:  z.string().nullable()
                   .describe('If requiresVector, an expanded clinical phrasing to embed.'),
});
// mapped to the Plan: useSql, useRag, needsSearch = useSql || useRag, semanticQuery
  • Four legal routes: SQL only · vector only · both · neither
  • "Neither" is a real route, not an error — "what's a normal A1C range?" has no tie to these records, so retrieval is skipped on purpose.
Frame it: yesterday's pattern pointed at a new target — instead of extracting urgency from an email, extract retrieval strategy from a question. The stub is in lib/agents/selector.ts; they write the system prompt + the select function (same four steps, history.slice(-5) so follow-ups like "what about her medications?" have a her).

Route, don't extract

Tempting: have the selector pull out "hypertension" and "under 50" while it's reading the question anyway. Resist.

A selector that only routes

Can only misroute. One failure mode — easy to inspect, easy to fix with a sentence in the prompt.

A selector that also extracts

A routing bug and an extraction bug per question. And the SQL specialist re-derives the entities better anyway — it holds the database schema; the selector extracts blind.

The one artifact it does write: semanticQuery — an expanded clinical rephrasing ("trouble sleeping" → "insomnia, poor sleep, waking at night"). Nothing downstream re-derives that, and the selector is the last LLM to see the question before it becomes a vector.

The test that separates the two: does any downstream component re-derive it better? Entities — yes (the SQL agent, schema in hand, next weekend). The semantic expansion — no; it's the selector's only real product besides the booleans.
