The problem: everyone sees everything

Your assistant answers real questions about patient records, and it works. Ask "look up Maria Gonzalez" and back comes her name, her birth date, her notes. Perfect — if you're her doctor.

But the front-desk scheduler runs the exact same query and gets the exact same chart. That's not a helpful assistant; that's a privacy leak with a chat box on top. A medical system must show identifying detail only to those allowed to see it.

Say it plainly: sit in the gap. The system already works — so why isn't it done? Because "works" and "safe to point at real people" are different bars. This week is the second bar.

The move: the server decides, from who you are

Two ideas do all the work this week:

  • Authentication — prove who is asking. A login mints a signed session; every request carries it.
  • Authorization — the server reads the role off that signed session and decides what the answer looks like. Not the client. Never the client.

Same question in, differently-shaped answer out — and the shape is chosen on the server, where the caller can't reach.

Who is this caller?

The system has had no humans in it all course. Now it gets a front door: log in, get a signed token, present it on every protected route.

          login (email + password)
                │  verify hash
        ┌───────▼────────────┐
        │ signed JWT (8h)    │  ← httpOnly cookie, page scripts can't read it
        └───────┬────────────┘
        every protected route:
        requireAuth(request)  →  a Session, or throw 401

One reusable guard — requireAuth in lib/auth.ts, built on jose. The token lives in a cookie the browser scripts cannot read, and never in the response body.

Land it: httpOnly matters — if JavaScript can read the token, one injected script steals the session. The cookie is the safe place precisely because the page can't touch it.

Two roles — and neither outranks the other

STAFF (front desk)

Books appointments. Sees pseudonyms, not names — no business reading the chart. Can schedule.

DOCTOR

Reads the full chart — real names, dates, note detail. Cannot book the calendar.

Each can do something the other can't. This is not a permission ladder where doctor > staff — a single "level 1, 2, 3" integer could never express it. Roles, scoped per action.

The trap to surface: students reach for a numeric permission level. Ask: which number is "can schedule but can't read charts"? There isn't one. That's why real systems use roles + per-action checks.

Same query, two shapes

"Look up patient 4c2a" — one endpoint, one code path, two outputs decided by the role on the session:

DOCTOR →  Maria Gonzalez · born 1975-04-12 · note: "reports feeling winded…"
STAFF  →  Patient-A7B3   · born 1975-XX-XX · note: identifiers scrubbed

STAFF still gets a useful answer — a stable pseudonym they can reference, a year for age math — just none of the identifying detail. The diff between those two lines is the whole lesson.

Say it plainly: role-shaped ≠ "STAFF gets an error." STAFF gets a real, working answer — de-identified. Denial and redaction are different tools; here we redact.
💬 Discussion

Given a role — what should they see, and how would you cheat it?

Half the room is STAFF, half is DOCTOR. For your role, answer three things:

  • What should you be able to see and do?
  • What must you be blocked from?
  • You're a malicious client — how would you try to cheat the boundary and pull data your role shouldn't have?
Facilitate: the cheat-attempt answer is the gold. Anyone who reaches for a client-sent role, a header flag, or "read it in the network tab" is reproducing broken access control in real time — name it, then show slide 9 answers it.

The role decides — the client can't opt out

Today obscurePII is client-controlled, which means it's not a security control at all. The fix is one line, on the server:

const session = await requireAuth(request);   // role from the SIGNED session
// STAFF are always obscured; a doctor may opt in
const shouldObscure = session.role === 'STAFF' ? true : clientRequested;

A STAFF caller sending obscurePII: false in the body and X-Obscure-PII: false in the header still gets Patient-A7B3. There's a test for each door, because a half-locked door is an unlocked door.

Land the principle: a control the caller can switch off is decoration. The role comes from a signed, server-issued session — a client can't mint a DOCTOR session without the server's secret.

The spec is the proof — run it

npm run test:run        # the RBAC specs go green

# lib/auth.test.ts              → token round-trips; 401 vs 403; tampered token → null
# app/api/query/route.test.ts   → STAFF forced obscured (body + header can't override)
# app/api/schedule/route.test.ts→ STAFF can book; DOCTOR POST → 403

We prove the role-shaped behavior through the specs, not a click-through — the tests attack from the body, the header, and a missing session, and pin every outcome. The scheduling gate is the inverse: DOCTOR is blocked, proving authorization is per-action, not a global rank.

Why specs, not the UI: there's no sign-in screen yet (no seed users), so the running app can't assume a STAFF vs DOCTOR identity. The specs already exercise both roles end to end — that gap is in the runbook, and closing it is a natural extension.

How the shaping happens: de-identification

When the role says "obscure," every patient object passes through lib/pii.ts before it leaves the server:

Structured fields

Name → stable pseudonym (Patient-A7B3, same name always same tag). Birth date → 1975-XX-XX. City/state → redacted.

Free-text notes

Regex scrubbing over the note body: names, SSNs, phones, emails, addresses → [REDACTED]. The clinical meaning stays; the identifiers go.

Pseudonymize what must stay referenceable; redact what doesn't. The pseudonym is a hash — consistent across responses, but it can't be reversed into the name.

Two principles under all of it

Minimum necessary

Each role gets exactly what its job needs — no more. The front desk schedules without ever reading a chart.

Accountable access

Every look at a record is attributable: who asked, for which patient, when. Redaction limits exposure; the log makes access reviewable.

These aren't invented for this app — they're the standard shape of handling sensitive records. You've been building the technical controls behind them all along.

Backstage: this is where the HIPAA connection lives — minimum-necessary access + audit trail are named requirements. Keep the law in the runbook; on the slide it's just "how you handle sensitive data." Never say "HIPAA-compliant."
🛠 Mini-challenge

Try to cheat the boundary — and watch it hold

  • As STAFF, send obscurePII: false in the body — confirm you still get Patient-A7B3.
  • Now try the X-Obscure-PII: false header — confirm both doors are locked.
  • Try claiming role: DOCTOR from the client — confirm the server ignores it (the role comes from the signed session).
  • Move the obscuring into the front-end instead, then read the real name off the raw response in the network tab — see why it must be server-side.
The point: if any of the first three succeed, you've found a real broken-access-control bug — fix it and add the test. The last one shows the browser is inside the threat model: redact before the boundary, or you haven't redacted.

Where you are — and where to go next

Week 1 you couldn't answer "which patients are short of breath?" because the database matched keywords, not meaning — so you built a second engine that searches by meaning. Five weeks later that system routes its own queries, calls tools, refuses what it shouldn't, knows who's asking, and shapes what they see.

You can build it, stress it, and defend every decision — including who's allowed to see a name. That's the difference between describing a RAG system and having built one.
