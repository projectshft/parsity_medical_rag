The problem: everyone sees everything

Your assistant answers real questions about patient records, and it works. Ask "look up Maria Gonzalez" and back comes her name, her birth date, her notes. Perfect — if you're her doctor.

But the same tool, wired into a front-desk assistant or an outside integration, hands over the exact same chart. That's not a helpful assistant; that's a privacy leak with a chat box on top. A medical system must show identifying detail only where the job needs it.

Say it plainly: sit in the gap. The system already works — so why isn't it done? Because "works" and "safe to point at real people" are different bars. This week is the second bar.

The move: the channel is the access level

Two ideas do all the work this week:

  • De-identification — a real function, lib/pii.ts, that turns a name into Patient-A7B3, a date into 1975-XX-XX, and scrubs identifiers out of note text.
  • The channel — which door a request comes through decides whether it gets full data or the de-identified view. No login, no roles, no client-sent access flag to forge.

Same question in, differently-shaped answer out — and the shape is decided by the entry point, on the server, where the caller can't reach.

Why not login+RBAC? That's the heavier tool for when one endpoint serves many human roles. Here the trust boundary lines up with the entry point — so modeling it as a channel is simpler and removes a whole class of forged-role bugs.

Two doors into the same data

Front-office channel — the MCP server

Reached by Claude Desktop / Cursor and outside tools. Exposes only non-identifying tools (search, notes, condition lists). Always de-identified — no off switch.

Clinician channel — the direct app

The chat UI and /api/query. Returns the full chart by default: real names, dates, note detail. May opt into obscuring.

Which door you came through is your permission. The front-office door only has de-identified tools — there's no patient-detail lookup on it to abuse.

The trap to surface: students reach for a client-sent "role" field. Ask: where would the server even read it from? There isn't one. The channel already encodes the trust level — that's the point.

How the shaping happens: lib/pii.ts

When a response must be de-identified, every patient object passes through lib/pii.ts before it leaves the server:

Structured fields

obscureName → stable pseudonym (Patient-A7B3). obscureDate → 1975-XX-XX. obscureLocation → redacted city/state/zip.

Free-text notes

obscureContent — regex scrub over the note body: names, SSNs, phones, emails, addresses → [REDACTED]. The clinical meaning stays; the identifiers go.

obscurePatient orchestrates all of it, field by field. Pseudonymize what must stay referenceable; redact what doesn't.

Pseudonym, not deletion — and why the year stays

obscureName("Maria Gonzalez")  →  "Patient-A7B3"   // SHA-256, first 4 hex
obscureDate("1975-04-12")      →  "1975-XX-XX"     // keep year, hide day
obscureContent("Call Mr. Smith at 617-555-0142")
                               →  "Call [NAME] at [PHONE REDACTED]"

obscureName is deterministic (same name → same tag, so records stay linkable) and one-way (you can't run the hash back to the name). The date keeps the year so age-based reasoning still works. A NON_NAME_WORDS set stops the name regex from mangling "Chief Complaint" or "Blood Pressure."

Land the distinction: a pseudonym is not anonymization — it's a stable, one-way tag still tied to a real row on the server. De-identification reduces exposure; it doesn't make the data anonymous.

The front-office default can't be switched off

On the front-office channel, obscuring is hard-coded — there's no argument a caller can pass to turn it off:

// mcp-server/index.ts — every tool, always
const result = await executeQuery(query, { sqlLimit: limit });
const formatted = formatResultsForLLM(result, true);   // ← always obscured

The direct clinician channel is the opposite: full by default, and the caller may opt into obscuring (obscurePII in the body, or the X-Obscure-PII header, which wins). That flag is a convenience on an already-trusted door — not the security boundary. The boundary is the front-office default nobody can reach.

Land the principle: a control the caller can switch off is decoration. The front-office channel is a real control precisely because it offers no switch.

The de-identification is real code — run it

npx vitest run lib/pii.test.ts   # the de-id layer, pinned

# obscureName        → same name → same Patient-XXXX; case-insensitive
# obscureDate        → "1985-03-15" → "1985-XX-XX"
# obscureContent     → SSN / phone / email / Dr.-name scrubbed…

Then open the two consumers: mcp-server/index.ts passes true to formatResultsForLLM on every tool (front-office, non-negotiable), and app/api/query/route.ts threads an optional flag (clinician, opt-in). Same lib/pii.ts, two channels.

Why the specs: the de-id tests mock nothing external and run on any network — they're the source of truth. Show one over-redaction case and one under-redaction gap; the imperfection is the honest part.

Regex de-identification is always imperfect

obscureContent is pattern-based, so it fails in both directions — and knowing which is the whole skill:

Over-redacts

A capitalized medical term looks like a name. That's why NON_NAME_WORDS exists — to keep "Blood Pressure" from becoming [NAME].

Under-redacts

An all-lowercase name, an unusual phone format, a novel identifier — the regex misses it, and a real name ships.

A redactor you trust blindly is worse than one you know the limits of. Naming one identifier your scrubber would still leak is exactly the honest sentence for your final video.

The principle under it: minimum necessary

Minimum necessary

Each channel exposes exactly what its job needs — no more. The front-office door schedules and searches without ever surfacing a real name.

De-identify at the boundary

Obscuring happens server-side, in the response, before it crosses the wire. The browser is part of the threat model.

This isn't invented for this app — "minimum necessary" is the standard shape of handling sensitive records. The channel model is one clean way to enforce it: the door that doesn't need PII never has it.

Backstage: this is where the HIPAA connection lives — minimum-necessary access is a named requirement. Keep the law in the runbook; on the slide it's just "how you handle sensitive data." Never say "HIPAA-compliant."
🛠 Mini-challenge

Try to cheat the boundary — and watch it hold

  • On the front-office (MCP) channel, hunt for any argument that makes a tool return a real name — confirm there isn't one.
  • On the clinician channel, send obscurePII: false in the body with X-Obscure-PII: true in the header — confirm the header wins.
  • Move the obscuring into the front-end instead, then read the real name off the raw response in the network tab — see why it must be server-side.
  • Feed obscureContent an all-lowercase name or an odd phone format — find one identifier it misses.
The point: if the front-office door ever leaks a real name, you've found a real bug — fix it and add the test. The network-tab one shows the browser is inside the threat model: de-identify before the boundary, or you haven't de-identified.
