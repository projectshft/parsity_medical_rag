The setup: it works — in your app

Three weeks in, the agent answers real questions: selector → SQL agent ‖ vector agent → one streamed answer. But it only lives in your chat UI. The front-desk staff don't want your app — they already live in Claude Desktop, and they want your system inside it.

Read that again: you're about to hand your patient database to a model you don't prompt, don't control, and can't see. Two questions decide everything this week: what may each caller see, and what happens when the AI acts on the world?

Sit in the discomfort: don't solve it yet. Someone will say "add a login" — park it visibly; the breakout returns to exactly that instinct and dismantles it. The hook for the session: one system, two callers with different rights, and by the end it doesn't just answer — it acts.

MCP: USB-C for AI

The Model Context Protocol: you implement one MCP server that exposes your system as tools; any MCP client — Claude Desktop, Cursor, a growing list — discovers and calls them at connect time. No per-client integration code, ever.

Claude Desktop ┐
Cursor         ├──▶  mcp-server/index.ts  ──▶  selector → SQL agent ‖ vector agent
any MCP client ┘     two tools: search_patients · query_notes

The client's LLM reads your tool names, descriptions, and schemas — and decides when to call what. Your tool descriptions are prompts for a model you don't control. The description is the interface.

Why not REST? They have API routes already — Cursor can't use them. REST serves programmers (a human reads docs, writes glue per client); MCP's contract is machine-readable at connect time. Honest cost: young protocol, and the transport is stdio — the client literally launches your server as a subprocess and talks over stdin/stdout. That one fact powers two of today's live failures. Keep this slide short; the wiring teaches it better.
💻 Code together

Wire it into Claude Desktop

One config file. The client launches your server as a subprocess and holds the stdio pipe open:

// ~/Library/Application Support/Claude/claude_desktop_config.json
{
  "mcpServers": {
    "medical-rag": {
      "command": "npx",
      "args": ["ts-node", "/ABSOLUTE/path/to/repo/mcp-server/index.ts"],
      "env": { "DATABASE_URL": "…", "OPENAI_API_KEY": "…", "PINECONE_API_KEY": "…" }
    }
  }
}
  • Absolute path — the subprocess inherits the client's working directory, not your repo
  • The env block — the subprocess inherits nothing; your .env doesn't travel
  • Full restart — the config is read at launch. Edit → Cmd-Q → reopen
The reveal: ask Claude, in plain language, "how many patients have high blood pressure?" — it announces a search_patients call, shows the arguments it chose, and waits for approval. A foreign model just picked your tool from your description alone. Live-failure notes: one stray console.log hangs the client (stdout IS the protocol channel); "I changed the config and nothing happened" = no full restart; tools erroring = missing env keys. All three are in the bank — and on the student branch, tool calls error with "Not implemented… lib/pii.ts" until the lab lands. That's the door demanding the lab; name it, don't fix it.

Which door you came through is your permission. No login, no roles, no per-request check — the server decides from the channel, and there's no role field on the request to forge.

Land the vocabulary: channel-based access vs role-based access. Same specialists behind both doors — the difference is one scrubbing line at the MCP door. Don't argue for it yet; the breakout does that work next.
💬 Discussion

No login on a patient database. Defensible?

  • The MCP door has no auth at all. Is that a hole — or a design? What exactly would a login protect?
  • When do channels beat RBAC — and what has to be true of the audiences for that to work?
  • Where does the channel model collapse? Name one tool, or one change, that would break it.
Facilitate toward the emptied room: you don't gate a room with nothing sensitive in it — you make sure nothing sensitive is in it. A permission system is a runtime promise (every tool must remember to check); a channel that cannot emit PII makes no promise it can break. Channels win when audiences split cleanly onto different entry points and one needs nothing sensitive; RBAC when one door must serve different entitlements to the same data. Collapse: get_patient "for convenience," or an obscure: false parameter. Full answer key in the runbook.

What makes the open door safe: two guarantees

1. A deliberately narrow tool set

Exactly two tools — search_patients, query_notes. No get_patient, no patient-detail lookup. The most sensitive query shape isn't denied — it's not offered.

2. Always-on obscuring

Every response is scrubbed before it leaves — names become Patient-A7B3, no flag to turn it off, no exceptions. Front-office channel, front-office data.

Break either guarantee and the whole design collapses. And guarantee #2 doesn't exist in your repo yet — lib/pii.ts is a stub. Time to build it.

The pivot line: "on my branch this works; on yours, 31 tests are failing and the scrub is a TODO. The tests are the spec — let's make them green."
💻 Code together — the PII lab

Implement lib/pii.ts: the tests are the spec

npx vitest run lib/pii.test.ts     # 31 failing — your job is zero
  • obscureName — SHA-256 of the lowercased name → Patient-A7B3. Deterministic (same name, same pseudonym — records stay linkable) and one-way (no running it backward)
  • obscureDate — keep the year, hide the rest: 1985-03-15 → 1985-XX-XX. "Patients over 60" still works; birthdays stop identifying
  • obscureContent — regex de-identification over free text: SSNs, phones, emails, addresses, Mr./Mrs./Dr. names, full-name patterns

Watch the over-redaction trap: a de-identifier that turns "Blood Pressure" into [NAME] is its own kind of broken — the tests pin that medical terms and vitals survive.

Build order = the test file's describe blocks: name → date → location → content → patient object. Each function flips its own block green; the room feels the progress. Full walkthrough and the likely failure modes are in the runbook.

Wrap it at the door — and why the scrub is shape-agnostic

The SQL side is text-to-SQL: the LLM writes the query and picks whatever columns it likes. There is no fixed "name field" to pseudonymize — so the channel scrubs the entire rendered text, whatever shape it took:

// mcp-server/index.ts — search_patients, the last line before text leaves
const combined = [sqlText, ragText].filter(Boolean).join('\n\n');
const formatted = obscureContent(combined);   // ← the front-office guarantee

// app/api/chat/route.ts — same specialists, NO scrub.
// The channel is the difference: one line per door.

Comment that line out "just to test" and the front-office door quietly becomes a PII leak — no error, no warning, nothing red.

Do the diff live: the same question through both doors — Patient-A7B3 · 1985-XX-XX vs the real name and birth date. The diff is the lesson. Then restart Claude Desktop and re-ask the question that errored earlier — now it answers, in pseudonyms, through code they wrote. Then hand them the hammer: next slide.
🛠 Break it

Your scrubber misses. Find where.

  • Feed obscureContent PII in a format the regexes don't expect — and watch it sail straight through
  • Try: an all-lowercase name · an SSN written with spaces · a non-US phone number · a DD/MM/YYYY date
  • For each leak: which pattern should have caught it, and why didn't it?

Regex de-identification is imperfect by design — a pattern list can't enumerate every way identity gets written. 31 green tests means the scrubber handles the formats we thought of. Knowing where it leaks is the skill.

Frame it precisely — a seed, not a bug: nobody "fixes" this by adding one more regex forever. The durable answer is measuring the leak rate — which is next week's whole subject. Don't teach evals now; just leave "the tests we thought of ≠ the inputs the world sends" ringing in the room.
🛠 Build — this week's deliverable 🎥

Add a tool of your own — and honor the contract

Every new tool on the front-office door inherits two obligations, or it breaks the channel for all of them:

  • Non-PII shaped — answers a front-office question (search, count, aggregate), never "hand me one named person's chart"
  • Obscured output — every name or identifier through obscureName / obscureContent before the text leaves
candidates:  list_conditions · compare_lab_across_patients · summarize_patient
prove it:    tools/list shows it  →  tools/call returns well-shaped text
             →  the text carries pseudonyms, never a real name

The proof is the deliverable: a client discovering and calling your tool, and obscured text coming back. Source code alone proves nothing about the channel.

Timebox: spec + skeleton in the room, finish at home. Shape: registerTool(name, { description, inputSchema }, handler) — copy the existing tools, return { content: [{ type: 'text', text }] }. The description is a prompt for a foreign model — the colleague test applies: can a fresh model route correctly from your descriptions alone?

Everything so far reads. Now the system acts.

Booking an appointment is the first write — and the moment an AI can act, one question outranks all others: does the AI act, or does the AI propose?

Weigh every action on two axes

Reversibility × cost of being wrong. A bad search costs a shrug and a rephrase. A hallucinated booking takes a real slot and corrupts a clinic's day.

The pattern: human-in-the-loop

The model proposes structured parameters. A human reviews and confirms. Deterministic code executes. The model never holds the trigger.

You've already stood on the other side of this gate: Claude Desktop asked your approval before every MCP tool call today. Now you build the gate yourself.

The scheduling flow: propose → approve → execute

"schedule Abe for Tuesday at 2"
        │
        ▼
detectSchedulingIntent        LLM extracts {patient, date, time} — structured output
        │
        ▼
findPatientByName             gate #1: the patient must EXIST — no match, no card
        │
        ▼
X-Scheduling-Action header ─▶ confirmation card — gate #2: a HUMAN reviews and edits
        │  confirm (posts the CARD's values, not the model's)
        ▼
POST /api/schedule ─▶ Cal.com books ─▶ Retell voice call confirms (best-effort)
Two beats to land: (1) two gates before a card even appears — intent must fire AND the patient must be real; a made-up name dies at findPatientByName, silently. (2) Confirm posts the card's current state — if it posted the model's raw extraction, the human gate would be decoration and the user's edits would vanish.
💻 Live demo

Book it for real — and listen

  • In the chat: "schedule Abe for tomorrow at 2pm" → the card appears with the extracted values
  • Adjust the time on the card. Confirm. /api/schedule books a real Cal.com slot
  • Then the room goes quiet: with DEMO_PHONE_NUMBER set, a phone rings — the AI is calling to confirm the appointment it just booked

The call is best-effort by design: wrapped so a failed call never undoes a successful booking. Reads fail cheap; writes get gates; notifications get retries-or-shrugs. Match the guarantee to the consequence.

The wow moment — protect it with pre-flight (Retell keys, agent deployed, DEMO_PHONE_NUMBER = your phone, whole loop tested this morning, ringer on, phone near the mic). If the call doesn't come: the booking still succeeded — show confirmationCall: { called: false } in the response and teach best-effort as the design, not a failure. Recovery script in the runbook.
🔍 Homework — research

Score your scrubber against the real standard

  • Read the HHS Safe Harbor list — the 18 identifiers that must go for data to count as de-identified
  • Audit obscureContent against all 18: which does it catch, which does it miss entirely?
  • Write the honest sentence: "my scrubber would still leak ___" — with a concrete example input
  • Skim MCP's security best practices — notice how much of "secure a server" reduces to be deliberate about what your tools can return

That honest sentence belongs in your final-week video. De-identification reduces exposure; it doesn't make data anonymous — and pretending otherwise is the real failure.
