Two questions you can't answer yet

Your system works — mostly. A doctor asks "which patients are short of breath?" and it answers. But two questions have no answer you can defend:

"Why did it answer that?"

The reply streamed out and you have no idea which notes it read, or which step went wrong. You can't console.log your way through an LLM.

"Is it any good?"

It feels right. But "feels right" isn't a number, and next week's change might quietly make it worse. You have no baseline.

Say it plainly: a non-deterministic system fails statistically and silently. Same input, different output, no stack trace. The rest of the week is the two instruments that fix that.

The move: two instruments

Observability — see the past

Wrap each step so a finished run is replayable. Open one answer and read exactly what happened: which step, how long, what text the model actually saw.

Evals — measure the present

A repeatable test with a number at the end. Change a prompt, re-run, see if the number moved. Judgment becomes measurement.

A trace tells you what happened on one run. An eval tells you whether it's good across many. You need both — one debugs, one decides.

First, how many LLM calls is "one answer"?

A single chat turn isn't one call to the model. Watch the agent handle one question:

  "which patients are short of breath?"
        │
   1 ─► analyzeQuery      → intent + which engines to use   (LLM call)
        │
        ▼  run SQL / vector search  (retrieval — no LLM)
        │
   2 ─► stream the answer → grounded prose for the user     (LLM call)

So ~2 LLM calls per turn — an analyzer and an answerer — plus an embedding call on the search path. Most people guess one. That gap is exactly why you need to see inside.

Land it: every one of those calls costs money and can fail on its own. If you can't see them, you can't debug them and you can't price them — the two threads of the whole week.

Observability: a debugger for the past

You can't set a breakpoint on a request that already finished and streamed to a user. So instead you record each step as it runs, and read the recording afterward.

Wrap a step in traced() and it reports the step's name, inputs, outputs, duration, and any error to LangSmith — where the run shows up as a tree:

execute_query                         820 ms
 ├─ analyzeQuery        (llm)         310 ms
 ├─ retrieval           (retriever)   120 ms
 └─ answer              (llm)         390 ms

Each box stores what went in and what came out. A trace is a witness to a run you can no longer watch live.

💻 Code walkthrough

The wrapper, and its two non-negotiable rules

export async function traced(name, fn, options) {
  if (!isLangSmithEnabled()) return fn();   // no key → run untraced

  const run = new RunTree({ name, ... });
  try {
    await run.postRun();
    const result = await fn();
    await run.end({ outputs: { result } });
    return result;
  } catch (error) {
    await run.end({ error: String(error) });
    throw error;               // ← re-throw the wrapped error
  }
}
  • Re-throw what you wrapped — record the crime, then let it hit the caller. Swallow it and you've hidden the bug you were trying to see.
  • Never throw your own error — no key, dead endpoint: the pipeline still answers. Observability watches; it never takes the request down with it.
Both from one idea: a trace is a witness, not a participant. It reports the crash; it is never the crash.
💻 Code walkthrough

Open one real run and read it

In LangSmith, click a hybrid query and walk the tree top to bottom:

  • analysis — intent + which engines fired. The first stack frame of every answer.
  • retrieval — how many patients and notes came back.
  • the rendered context — the exact text handed to the model. Read it aloud; this one field pays for the whole setup.
  • timing — which step ate the wall-clock (usually an LLM call, not the database).
The 30-second fix: "the AI hallucinated" almost always becomes "retrieval handed it the wrong context" — and the rendered-context field shows that instantly. Count the LLM boxes: it's 2.
💬 Discussion

The answer was wrong. Where do you look first?

A user asked about one patient's diabetes and got a confident answer about someone else's. You have the trace open. Walk it:

  • Wrong intent or entities in the analysis step? → the fault is upstream, in the analyzer prompt.
  • Analysis right, but the rendered context holds the wrong patient's notes? → retrieval / filtering.
  • Context right, but the answer ignores it? → the grounding prompt.
Facilitate: drive to the habit — read the trace top-down, don't guess. Without the trace, every one of these looks identical from the outside: "it lied." The trace tells you which door to open.

Evals: the spine of the whole course

A trace explains one run. It can't tell you whether the system is good. For that you need the rule this course keeps coming back to:

No metric, no decision. You don't argue about whether a change helped — you measure it, and let the number decide.

An eval is embarrassingly simple machinery: a fixed set of inputs, a known-right answer for each, and a score at the end. Its value isn't today's number — it's the next time you tweak a prompt and a number you weren't watching moves.

Say it plainly: this is the difference between an engineer who built a system and one who finished a tutorial. You can show the number that made you keep a change — or drop it.

Kind 1: exact match — grade the classifier

The analyzer is a classifier: it returns an intent and two booleans. Classifiers get the most satisfying eval there is — the answer either came back right or it didn't. Compare to labels you wrote first, by hand:

const a = await analyzeQuery(c.query);
intentOk += a.intent          === c.expectIntent  ? 1 : 0;
sqlOk    += a.requiresSQL      === c.expectSQL     ? 1 : 0;
vectorOk += a.requiresVector   === c.expectVector  ? 1 : 0;
// example run → intent 84%   SQL 96%   vector 92%   (n=25)
// (illustrative — your cases, your numbers; the point is you HAVE numbers)

Score the booleans separately from intent — they're what the router actually branches on. No second model, no judgment call: just ===.

Kind 2: LLM-as-judge — grade an answer

Exact match works when there's one right answer. But "is this answer grounded in the notes?" has no string to compare against. So you ask a second model to grade the first:

evaluateAnswerFaithfulness(context, answer)
  → { score: 9, pass: true, reasoning: "every claim traces to the notes" }

evaluateRetrievalRelevance(query, retrievedNotes)
  → { score: 3, pass: false, reasoning: "notes are about a different visit" }

A judge prompt with a 0–10 rubric, temperature: 0, threshold at 7. It's noisier than === and it costs a call — so you read it as a trend over runs, not a hard pass/fail gate.

Pre-empt the smirk: "using an LLM to grade an LLM?" Yes — grading is easier than generating, the judge sees the rubric, and you spot-check it against your own eyes. It's a thermometer, not a supreme court.

Retrieval eval: hit@5, scored twice

Back to the search itself. Build 15+ (query, expected-note) pairs from your own corpus — the query in a user's words, the note you know should surface. Then:

hit@5 = (queries where the expected note lands in the top 5) / total

   vector alone      → 73%
   vector + reranker → 89%      (illustrative numbers)

"Is 73% good?" The only honest answer is: compared to what?

An eval doesn't hand you a grade — it hands you a difference. Two configurations, two numbers, and now the reranker's value is a fact, not a vibe.

Land it: plenty of real teams measure and drop the reranker; plenty keep it. The eval is what tells you which kind your system is. Keep a few pairs you never tune on.

Cost is a metric too

You now measure quality. You've never put a number on cost — and an LLM system with no cost number is one that surprises you with an invoice.

The trace already counts the calls. Price them:

per chat turn:  2 LLM calls (analyzer + answerer) + 1 embedding
                × tokens in/out × model price  →  a per-query $ figure

Now a real decision lands with both numbers. A reranker that adds 16 points of hit@5 and triples the per-query cost isn't a default you flip on — it's a trade you make with the accuracy gain and the price tag side by side.

Say it plainly: "no metric, no decision" applies to cost too. The teams that get cost-surprised in production are the ones who never made it a metric.
🛠 Mini-challenge

Run the evals — then break them

  • Run npm run test:evals. Read the retrieval + judge numbers — that's your baseline. Write them down.
  • Break the reranker silently: unset COHERE_API_KEY and re-run. Search still "works" — but reranked order now equals plain vector order, no error shown. Did your hit@5 quietly drop?
  • Blind the trace: unset the LangSmith key. The app keeps answering; the trace project goes silent. You went blind without going down — on purpose.
The discipline: a failure you can measure is a failure you can defend against. The silent-fallback one is the crowd favorite — nothing the user sees says reranking stopped. Only the eval catches it.
🔍 Research

Go find out — and where you are

  • Your labels are the ground truth — you invented them. So how is grading against your own labels legitimate, and not circular? Where's the line between this and cheating?
  • Deterministic tests gate every commit; LLM evals don't. Why is a paid, noisy metric a bad commit gate but a good thing to watch over time?

Where you are: you can see why the system answered (the trace) and prove whether it's good (the eval) — including what it costs. Judgment is now measured, not felt.
