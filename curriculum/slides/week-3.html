The problem this week solves

Everything you've built answers questions through your chat box. But the people you'd build this for already live in other AI tools — Claude Desktop, Cursor, a dozen assistants.

Do you write a separate integration for each one? Or expose your system once, in a way any of them can discover and use?

Say it plainly: the question isn't "how do I add a feature." It's "how does my system become something other software plugs into."

MCP: USB-C for AI

The Model Context Protocol is an open standard for connecting AI apps to outside systems. You write one MCP server that exposes your RAG as tools; any MCP client can discover and call them.

  Claude Desktop ─┐
        Cursor ───┼──▶  YOUR MCP server  ──▶  your pipeline
  any MCP client ─┘     (mcp-server/index.ts)   (search, notes, patient detail)

A REST API needs a human to read docs and write glue code per client. MCP's contract is machine-readable: the client discovers your tools, schemas and all, at connect time. REST serves programmers; MCP serves models.

Land the one line: write one server, any client discovers your tools. The difference from REST is discovery — a model reads the contract, not a programmer.

A tool description is a prompt for a model you don't control

server.tool(
  'query_notes',
  'Search clinical notes using semantic search. Use this for finding
   relevant medical notes, symptoms, treatments, or observations.',
  { query: z.string().describe('e.g. "chest pain", "breathing problems"'),
    patientId: z.string().optional(),
    topK: z.number().optional().default(5) },
  async ({ query, patientId, topK }) => { /* call your pipeline */ }
);

The client's LLM reads your name + description + parameter schema and decides when to call it. Vague description → wrong tool at the wrong time — and you can't fix it by editing their prompt. The description is your interface.

💻 Code walkthrough

Talk to the server before any AI does

# the server speaks JSON-RPC over stdin/stdout — pipe to it, no AI required
echo '{"jsonrpc":"2.0","id":1,"method":"tools/list","params":{}}' \
  | npx ts-node mcp-server/index.ts

Back comes a JSON listing of your tools with their schemas — proof the contract is discoverable. This is exactly what Claude Desktop sees at connect time. One rule that bites everyone: never console.log in this server. Stdout is the protocol channel; a stray log line corrupts the stream and the client disconnects. Log to console.error instead.

💻 Code walkthrough

Wire it into Claude Desktop, watch a foreign model use it

// ~/Library/Application Support/Claude/claude_desktop_config.json
"mcpServers": {
  "medical-rag": {
    "command": "npx",
    "args": ["ts-node", "/ABSOLUTE/path/to/mcp-server/index.ts"],
    "env": { "DATABASE_URL": "...", "PINECONE_API_KEY": "...", "OPENAI_API_KEY": "..." }
  }
}

Restart fully. Ask in plain words — "how many patients have high blood pressure?" — and watch it pick a tool, show its arguments, and wait for your approval. The path must be absolute (relative paths are the #1 failure). The client launches your server as a subprocess — crash and the tools vanish. And notice: you just pasted database credentials into a desktop app's config file.

💬 Discussion

You shipped a patient database with no front door

Anyone who can launch that server has unrestricted access to every tool — no identity, no permissions, no record of who asked what.

  • What's the worst thing a stranger could do with your tools right now?
  • Which credentials are sitting in plaintext, and where?
  • For medical data, is this a missing feature — or a missing wall?
Facilitate: don't rescue them yet. Let the discomfort land — that discomfort is the securing segment. Breakout if >8 people; debrief with the answer key.

"Secured" means three layers

Authentication — who's calling?

API keys. The caller presents one; the server validates it. The caller is a process, not a person — so keys, not logins.

Authorization — what may they do?

Scopes on the key, checked per tool: read (search), read_pii (named patient detail), admin (bulk listing).

Audit — who did what, when? A log entry for every call, allowed or denied. The modules are provided (mcp-server/auth.ts, audit.ts); you wire them in through one wrapper.

Least privilege: one key per client, smallest scope that works

Two callers, same system. A read+read_pii key can search and pull a patient chart — but is denied the admin-only bulk-listing tool. An admin key runs the same call and gets the data.

list_patients_by_condition  ("high blood pressure")

  read + read_pii key  →  DENIED: key lacks 'admin' scope   ← clean text, not a crash
  admin key            →  returns the list                  ← same call, two worlds

When a key leaks — and keys leak — the blast radius is the scope, not the database. And the key is revocable on its own, without rotating your infrastructure.

Design point to land: wire auth through one withAuth wrapper, not tool-by-tool. The next tool you add ships protected because it can't skip the wrapper.

The moment a system can act: does it act, or propose?

Reads are safe. Now the system gets its first write — booking a follow-up appointment. So the design question that outranks all others:

"schedule Smith ──▶ detect intent ──▶ confirmation card ──▶ confirm ──▶ book it
 for Tue at 2"     (LLM, structured)   (human reviews)    │        (code, deterministic)
                                                          └─ cancel ─▶ nothing happens

The model never holds the trigger. Its authority ends at proposing structured parameters; a human reviews the actual values and clicks confirm; only then does deterministic code act.

💻 Code walkthrough

Propose in the model, act in the code

// lib/scheduling.ts — the model only PROPOSES structured values
const SchedulingIntentSchema = z.object({
  isSchedulingRequest: z.boolean(),
  patientName: z.string().nullable(),      // null, never a guessed name
  suggestedDate: z.string().nullable(),    // "next Tuesday" resolved to YYYY-MM-DD
  suggestedTime: z.string().nullable(),
});
// → renders a confirmation CARD the human edits and confirms
// → only then does POST /api/schedule call the calendar (lib/calendar.ts)

The card's values — the ones a human just reviewed — are the source of truth, not the model's raw extraction. Dates get resolved to a real YYYY-MM-DD in code, because "next Tuesday" needs today's date and the model doesn't reliably know it.

💬 Discussion

Which actions are safe to automate — and which need a human?

Sort these onto two axes: reversibility and cost of being wrong.

  • Search a patient's records
  • Book an appointment on a real calendar
  • Send a refill request to a pharmacist
  • Send a referral letter under a doctor's name
Facilitate: the question is never "can the model do it" — it's "what does wrong cost, and who absorbs it?" That's what decides the gate. Debrief with the answer key.
🛠 Mini-challenge

Secure a tool, then try to break it

  • Prove the wall: call the admin tool with a read key → watch it refuse; swap in the admin key → watch it succeed. Same call, two callers.
  • Prove the record: find that denial in the audit log — grep "who tried to access what, and were they denied?"
  • Attack the gate: make "confirm" post the model's values instead of the card's. Watch the human's edits vanish. Now the gate is decoration.
Land this: don't show it working — show it refusing, recording, and recovering. The refusals are the deliverable, not the happy path.
🔍 Research

Go find out

  • Your read key can't pull a named chart — but it can walk the whole database politely, one call at a time. Auth doesn't stop that. What does? (Look up rate limiting and anomaly detection.)
  • "next Tuesday" needs today's date, and the model doesn't reliably know it. Why resolve dates in code, not in the prompt — even though the model can usually do the math?

Bring one finding back. The first question is a door nobody guards yet — you'll meet an attacker at it in the final week.

Where you are

  • And one human-gated write path — the model proposes, a person confirms, code acts
