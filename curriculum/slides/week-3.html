Foundations recap

  • An LLM is a next-token predictor. It answers by predicting what comes next — which is why it can read records you hand it, but can't search them on its own.
  • The missing piece is search by meaning. That's what we build this week — and it runs entirely on turning text into numbers.

A vector is just a list of numbers — a location

Turn a piece of text into a list of numbers and you've placed it at a point in space. Do it for every note, and similar meanings land near each other:

   "trouble breathing"  •   • "shortness of breath"
                                 • "dyspnea"

                                              • "quarterly earnings"

Our model returns 1,536 numbers per text — a point in 1,536-dimensional space. You can't picture 1,536 dimensions; two is enough for the intuition. How we measure "near" comes next.

The promise we've been deferring

Since Week 1 we've said "a search that understands meaning." A user types trouble breathing; the note only says shortness of breath. Zero shared words — and we promised the system would still find it.

No synonym table did that. So what does? Today the curtain comes up: embeddings.

Say it plainly: the hand-maintained synonym dictionary from Week 1 was the stand-in. This week we replace it with geometry learned from data.

An embedding turns text into a point in space

One sentence: an embedding turns text into a list of numbers, so that texts with similar meaning get nearby lists. Our model outputs 1,536 numbers per text — a point in 1,536-dim space. Picture two dims instead:

   shortness of breath  •   • difficulty breathing
            dyspnea  •                      • knee pain
                                                 • quarterly earnings

   search = embed the query, then rank docs by closeness:
   cosine(a, b) = (a · b) / (|a| × |b|)    └dot product┘ ÷ └lengths┘

Cosine asks "do these point the same way?" — 1.0 for identical down toward 0 for unrelated. It's the dot product over the two lengths; since these vectors are unit-length, the dot product alone gives the same ranking.

Design point to land: the geometry is the synonym dictionary — learned from data, not hand-typed.
💻 Code walkthrough

Measure the ladder yourself

Embed five phrases, score them against "shortness of breath" with a ten-line cosine function — real output:

dimensions: 1536
SOB vs dyspnea:    0.701   ← synonyms, zero shared words
SOB vs difficulty: 0.743
SOB vs knee pain:  0.327   ← same domain, different topic
SOB vs earnings:   0.163   ← unrelated

Synonyms ~0.70, same-domain ~0.33, unrelated ~0.16. The zero-shared-keyword pair scored 0.701 — that number is the entire reason the second half of this system exists.

Now watch it lie

Re-run with full sentences instead of bare phrases:

"the patient reports shortness of breath"  vs
   "dyspnea on exertion"        →  0.597
   "the patient reports knee pain"  →  0.622   ← !!

The knee-pain sentence beats the real synonym. Why? The two sentences share the template "the patient reports…". Embeddings measure overall textual similarity — meaning dominates, but phrasing and boilerplate leave fingerprints too.

Plant it: not a bug to fix today. Two later pieces — filtering before the geometry, a second opinion after it — exist for exactly this.

💻 Code walkthrough

Be the vector database — by hand, first

Before touching Pinecone, build what one does in ~ten lines: embed a tiny corpus once, embed a query, score, sort.

const docVectors = await createEmbeddings(corpus);   // text → vectors, once
const queryVector = await createEmbedding(query);
const ranked = corpus
  .map((text, i) => ({ text, score: cosine(queryVector, docVectors[i]) }))
  .sort((a, b) => b.score - a.score);                 // rank by similarity

Query "caring for the people around you" ranks "Love your neighbour as yourself" first — zero shared words. That map → sort → take top loop is what a vector database does. No extra magic in the box.

So why pay for a vector database?

If search is just that loop — two reasons, and only two:

Persistence

Your script re-embeds the corpus every run. A database embeds once and keeps the vectors — you pay the embedding cost a single time.

Speed at scale

Your loop checks every vector — fine for 5, hopeless for 144,000 per query. The database builds an index that finds nearest neighbors without checking them all.

Land it: everything else — metadata filtering, the API, the dashboard — is convenience around those two. Two index params are permanent: dimensions 1536 (must match the model) and metric cosine.

One new idea: filter inside the search

Pinecone can restrict the nearest-neighbor search to vectors whose metadata matches — searching one patient's 40 notes instead of all 144,000.

filter: { patientId: { $in: ['abc', 'def'] } }

This is not filtering afterward. A global top-10 trimmed to one patient might leave zero; a filtered search returns that patient's true top 10 — and the other patients' note text never leaves the database.

Design point to land: patientId is the privacy boundary, and this filter is where it's enforced — quality reason and safety reason.

Hybrid: facts narrow the world, meaning ranks what's left

"Sleep problems in patients with depression." Neither engine alone: SQL's LIKE '%sleep%' misses "insomnia"; vector search finds sleep notes for anyone. Run them in order:

  question
     │
  1. Postgres:  who HAS depression?  ──► patient ids   (exact)
     │
  2. Vector search: "sleep problems", filter to those ids   (fuzzy)
     │
     ▼
  notes about sleep, only from depressed patients

Exact filters narrow the world; semantic search ranks what's left. The exact step runs first — the math is quiet otherwise, and a fact is a WHERE, not a similarity score.

🛠 Mini-challenge

Prove the hybrid earns its keep

  • Run the depression + sleep hybrid. Now run the vector-only control: same query, no filter.
  • Count: how many of the global top-10 are not depression patients?
  • That count is the hybrid's value — quantified in thirty seconds.
  • Watch the empty-filter trap: if no patients match, an empty patientIds array means no filter → it searches everyone. That's a privacy bug, not a relevance bug.
The point isn't the query — it's the control. Plausible-looking results are the most dangerous kind; you beat them with counts, not by staring harder.

When cosine lies: a second opinion

Remember the lie — knee pain out-scored dyspnea. The root cause: an embedding summarizes each text separately, before any query exists. At search time you're comparing two blind summaries.

A reranker reads the query and the document together, at query time, and scores how well this doc answers this query. Boilerplate matching boilerplate impresses it far less.

query ─► vector search ─► top 25 ─► reranker ─► top 5, re-ordered
         (fast, broad)            (slow, careful)

Cheap-and-broad finds candidates; expensive-and-careful orders them. Same logic as skimming 200 résumés and interviewing 6.

💻 Code walkthrough

Wire the funnel — and resist concluding anything

const candidates = await searchChunks(query, 25);   // over-fetch on purpose
const reranked  = await rerankResults(query, candidates, 5);
  • Over-fetch is the point — a great note ranked #19 by cosine is invisible unless the funnel mouth is wide enough.
  • Different universe — rerank scores aren't cosine scores; only order within one list means anything.
  • The order changes… sometimes. Which raises the honest question.
Heads up (it'll bite live): no COHERE_API_KEY → the wrapper returns the original order, no error. If reranked == original always, suspect the key before the model.

"It felt better" is not evidence

You watched the order change and it felt like an improvement. But "the order changed" and "the results got better" are different claims — and the second needs ground truth: which notes are actually relevant? Nothing you've built knows that yet.

The course's spine rule arrives: no metric, no decision. A reranker adds latency, cost, and a vendor. Whether it pays for itself on your corpus is an empirical question — and you're one eval away from the answer.

Say it plainly: plenty of real systems measure and drop the reranker; plenty keep it as their best upgrade. Today we build the instrument that tells you which kind yours is.
🛠 Mini-challenge

Build day: your retrieval eval set

Embarrassingly simple machinery — and the most important build in the course so far:

  • 15+ (query, expected) pairs — questions with known-correct answers in your corpus, read from your own notes
  • Query in the user's words, not the note's (verbatim queries test nothing)
  • Metric: hit@5 = (queries where an expected note landed in the top 5) / total
  • Score it twice: vector alone vs vector + reranker. Now you know.
Land it: "is 73% good?" → the only honest answer is compared to what? Evals create differences, not grades. Keep a few pairs you never tune on.
🔍 Research

"Hybrid" can mean two engines — or two scores

Search "hybrid search" outside this course and you'll mostly find a different technique than the one you built:

Two engines (ours)

Postgres for exact facts, the vector index for meaning — picked per question.

Two scores (the literature)

Dense (meaning) + sparse/BM25 (exact words) fused in one store, because each catches what the other misses.

Go find out: why does this system get its exact-match half from Postgres instead of sparse vectors? Where's the one seam where sparse would earn its place? (Hint: a term that lives only in free-text notes, never as structured data.)
