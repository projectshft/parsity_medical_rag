The problem RAG solves

Ask any LLM "what medications is this patient taking?" — it has no idea. The records were never in its training data, and never will be.

The fix is almost embarrassingly simple: fetch the relevant records at question time and paste them into the prompt. The model doesn't need to know your data — it needs to read it.

Say it plainly: the hard part isn't the "generation." It's the retrieval — and retrieval is a search problem, which is where the engineering lives.

Two kinds of questions → two engines

Structured

"How many patients have diabetes?" An exact answer — a COUNT(*). This is a database job.

Semantic

"Patients whose notes mention trouble sleeping." No shared keywords with "difficulty falling asleep" — same meaning. A meaning-based search job.

Most real questions are one — or a hybrid of both. That's why the system has two storage engines.

💬 Discussion

Sort these five into structured / semantic / hybrid

  • "How many patients are on insulin?"
  • "Which patients describe chest pain at night?"
  • "Do any diabetics mention medication side effects in their notes?"
  • "Summarize this patient's health history."
  • "What's a normal blood pressure?" (trick — should this even be answered?)

# Chief Complaint
No complaints.

Two things to point out live: it has structure (headed sections) and it's short (~450 chars). That second fact decides a major architecture choice later — and it's a measurement, not a vibe.

The structured half: Postgres + Prisma

One schema file becomes both the database tables and a typed client.

  • Patient — name, birth/death date, phone, location
  • Condition — "Type 2 Diabetes Mellitus", status, onset
  • Observation — labs & vitals with numeric values
  • Medication — name, status, dosage
Design point to land: the FHIR id is the primary key → re-running ingestion updates a patient instead of duplicating. That property carries all the way to Week 6's upload API.
💻 Code walkthrough

Load the structured data, look at it

npm run db:push                    # create the tables
npm run ingest -- --limit 50 --skip-vectors   # Postgres only
npm run db:studio                  # browse the rows

--skip-vectors loads only Postgres — we haven't built the search half yet. Open Studio live and show real patients/conditions landing.

"diabetes" isn't in the data — "Type 2 Diabetes Mellitus" is

A naive WHERE display = 'diabetes' returns nothing. Real medical vocabulary is messy.

diabetes: ['diabetes', 'diabetic', 'diabetes mellitus',
           'type 2 diabetes', 'prediabetes']

One user word → many database phrasings. Blunt, hand-maintained — and it works because someone wrote the synonyms down. Contrast this with what the meaning-based search does automatically (Week 3).

🛠 Mini-challenge

Add a condition the dictionary is missing

  • Pick a condition not yet mapped (asthma, COPD, depression…)
  • Query the data first — what does it actually call this?
  • Add the real phrasings; re-run the count
  • Record the before/after number. "It seems better" is not done; "matched 0, now matches 14" is done.
The point isn't SQL — it's the loop: find a gap → fill it → verify with a number. That's the whole course in miniature.
