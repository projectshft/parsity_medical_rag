The setup: the data is already here

You join a clinic's engineering team. Years of records are already in Postgres — patients, diagnoses, medications, labs, and ~144,000 free-text clinical notes. Nobody needs you to build the database. It exists.

What they want: let a doctor ask "which patients are short of breath?" and actually find them — including the note that says "dyspnea on exertion."

Say it plainly: we're not teaching them to store data — they have that. We're teaching the thing on top: search by meaning. That reframing is the whole course.

Why the database alone can't do it

The obvious move is to search the notes column:

SELECT * FROM notes WHERE content LIKE '%shortness of breath%';
-- 0 rows.  The note says "dyspnea on exertion."

Keyword / SQL search

Matches letters. "shortness of breath" ≠ "dyspnea" ≠ "winded climbing stairs" — zero shared words, so zero matches.

What we need

Match meaning. All three phrases describe the same clinical fact. Find the note regardless of the words used.

Facilitate: ask the room for three more phrasings of the same symptom. The point lands when they generate the misses themselves.

The fix: turn text into a point in space

An embedding turns a piece of text into a list of numbers — a point in space — so that texts with similar meaning land near each other. Picture two dimensions:

   "shortness of breath"  •   • "dyspnea on exertion"
                                  • "winded climbing stairs"

                                                • "broken ankle"

Our model returns 1,536 numbers per note — a point in 1,536-dimensional space. You can't picture that; two is enough for the intuition. "Near" is the whole game — next we make "near" precise.

How "near" is measured: cosine similarity

Two texts are similar if their vectors point the same direction. Measure the angle between them:

similarity = cos(θ) = (A · B) / (|A| × |B|)
             └ dot product ┘   └ their lengths ┘

  1.0  → same direction  (nearly identical meaning)
  0.0  → unrelated

The dot product over the two lengths. Since these embeddings come back length-1, the dot product alone gives the same ranking. That's the entire engine under "search by meaning."

Design point to land: they should leave able to say what cosine measures in one sentence — direction, not distance-from-origin. This is the "understand it, don't black-box it" beat.
💻 Code walkthrough

Be the vector database — by hand

Before any vector store, do what one does, in ten lines: embed a tiny corpus, embed a query, rank by cosine.

const docs = ['dyspnea on exertion', 'broken ankle', 'well-controlled diabetes'];
const q = 'patient is short of breath';

const [qv, ...dv] = await embed([q, ...docs]);
docs
  .map((d, i) => ({ d, score: cosine(qv, dv[i]) }))
  .sort((a, b) => b.score - a.score)
  .forEach(r => console.log(r.score.toFixed(3), r.d));
// 0.71  dyspnea on exertion   ← no shared words, still #1
// 0.18  broken ankle

That loop is a vector database. A real one just does it fast, over millions of vectors, and stores them for you.

💬 Discussion

Which of these needs meaning — and which is just a lookup?

  • "How many patients have diabetes?"
  • "Which patients mention trouble sleeping?"
  • "Notes that describe chest pain at night"
  • "List patients over 65"
  • "Anyone whose notes sound like they're struggling to cope?"
Facilitate: drive to the split — counts/filters are exact (the DB already does them); "sounds like / describes / mentions" is meaning (needs vectors). That split becomes next week's routing agent.

Two stores, one source of truth

   Postgres  ──(vectorize)──▶   Vector store (Pinecone)
   system of record             derived search index
   EVERYTHING:                  just the notes + metadata,
   patients, meds, labs,        embedded so you can search
   notes (full text)            them by meaning

Postgres stays the source of truth. The vector store is a derived copy of the notes, built for one job: semantic search. If they disagree, Postgres wins — you just rebuild the index.

Say it plainly: this is the mental model that survives the whole course. The vector DB is never the truth; it's a searchable projection of the truth.
💻 Code walkthrough

The vectorize script: DB → vector store

Read the notes from Postgres, embed them, upsert them into the vector store. This is how you "service" existing company data for semantic search.

const notes = await prisma.note.findMany({
  include: { patient: { select: { firstName: true, lastName: true } } },
});

const chunks = notes.map((n) => ({
  id: n.id,                       // reuse the note id → idempotent re-runs
  content: n.content,
  metadata: { patientId: n.patientId, type: n.type, date: n.date },
}));

await upsertChunks(chunks);        // embeds + upserts in batches
npm run vectorize -- --limit 200   # a cheap slice while you build

Metadata is what makes the index useful

A bare vector answers "what's similar?" — but real questions are scoped: this patient, recent notes, a note type. So every vector carries structured tags copied from Postgres:

  • patientId — filter a search to one patient's notes
  • type, date — "recent H&P notes," not all notes ever

Choosing the right metadata is a real design decision — it decides which questions you can answer. That's your homework prompt: look at the data and propose what to tag.

Design point to land: one patient has ~113 notes on average (up to 2,162). Without patientId on the vector, "summarize this patient" is impossible. Metadata isn't decoration.
🛠 Mini-challenge

Search it — and try to make it miss

  • Vectorize a slice, then search: "patient struggling to breathe" — did it find the "dyspnea" notes?
  • Now try to break it: a query so vague everything matches a little ("patient has a problem"). Watch the scores flatten.
  • Filter by patientId — same query, one patient. Confirm no one else's notes leak in.
The point isn't the search call — it's building the instinct to interrogate results: what did it rank #1, and does that make sense? "Looks right" isn't a measurement; we make it one later.

One more question: what is "one piece"?

You just embedded each note as a single vector. That worked because a clinical note is already the right size — one encounter, one topic, ~450 characters, self-contained. One note = one piece. No splitting needed.

Say it plainly: we got lucky — our data came pre-cut into perfect pieces. That is not normal. Most corpora don't.

…but most text isn't so tidy

Imagine the opposite: one enormous document — a 4-million-character book. Embed the whole thing as one vector and it "means" everything and matches nothing. So you must split it — but where? Mid-sentence? Every paragraph? By chapter?

That splitting decision is chunking, and it's a craft. You'll learn it on a corpus that truly needs it — the King James Bible — as this week's side project.
